GRE Tunnels + OSPF over WAN
Cisco Packet Tracer lab project: two branch offices (branch1, branch2) connect to a head office through a provider (ISP) using GRE tunnels, with a single OSPF area 0 running inside them. Each branch has a redundant L3 core (HSRP) and an access layer with basic security hardening (port security, DHCP snooping, ARP inspection).
Topology
```
                    \[head-office] ---- 70.70.70.0/24 ---- \[Server4] (NAT outside)
                         |
                    85.85.85.0/30
                         |
                       \[ISP]
                      /        \\
            85.85.85.4/30    85.85.85.8/30
                /                    \\
          \[branch1 router]      \[branch2 router]
            /          \\           /          \\
      core-1-br1    core-2-br1  core-1-br2  core-2-br2
       (HSRP)         (HSRP)     (HSRP)       (HSRP)
          |              |          |            |
      access switches (VLANs)   access switches (VLANs)
```
GRE tunnels run from the branch routers through the ISP to the head office:
Tunnel	Branch	Tunnel address	Source / Destination
Tunnel1	branch1 → head-office	10.10.10.5/30 ↔ 10.10.10.6/30	85.85.85.9 ↔ 85.85.85.1
Tunnel1	branch2 → head-office	20.20.20.1/30 ↔ 20.20.20.2/30	85.85.85.5 ↔ 85.85.85.1
MTU on all tunnel interfaces is set to 1476, accounting for the 24-byte GRE overhead on top of the standard 1500.
Routing
OSPF area 0 — a single area spanning both the GRE tunnels (WAN) and the branch-internal links (between core and access via SVIs).
All VLAN interfaces (SVIs) on the core switches are configured as `passive-interface` — no Hello packets are sent out, but the routes are still advertised.
`head-office` injects a default route into OSPF (`default-information originate`), acting as the sole internet exit point.
A local static route (`ip route 0.0.0.0 0.0.0.0 ...`) on every device points to the next hop toward the ISP/head-office as a fallback.
VLAN layout
Branch	VLAN ID	Purpose
branch1	50, 60, 70	user segments (core1)
branch1	80, 90, 100	user segments (core2)
branch1	201	management
branch2	10, 20	user segments (core1)
branch2	30, 40	user segments (core2)
branch2	200	management
all	876	native VLAN on trunk ports
Each core switch holds active HSRP (priority 150 + preempt) for "its own" VLANs and acts as backup for the neighboring core's VLANs — a classic active/active load-balancing setup between the two L3 switches at each branch.
Link redundancy
Port-channel (LACP, `channel-group ... mode active`) between the access and core layers at both branches — protects against a single physical link failure.
Two independent core switches at each branch with cross-connected trunk links to the access switches.
`spanning-tree guard root` on ports facing the access layer — the core always stays the root bridge.
Security
`service password-encryption`, `enable secret` (hashed passwords).
SSH v2 only (`ip ssh version 2`, `transport input ssh` on vty), no Telnet.
`login block-for` — brute-force protection for SSH/console login.
Port security with sticky MAC on end-user access ports.
DHCP snooping + Dynamic ARP Inspection on branch2's access switches (not configured completely because in packet tracer i cant make etherchannel port trusted).
NAT (PAT/overload) on head-office for branch internet access via 70.70.70.0/24.
Known limitations and possible improvements
NAT on tunnel interfaces is not configured — a Packet Tracer limitation: `head-office` has `ip nat outside` and an ACL for `ip nat inside source list 100 ... overload`, but `ip nat inside` cannot be applied to Tunnel-type interfaces — Packet Tracer doesn't support this command on GRE tunnels (unlike real IOS). On real hardware, simply adding `ip nat inside` on Tunnel0 and Tunnel1 would be enough.
`sl\_def\_acl` — auto-generated by Cisco IOS when `login block-for` is configured, not bound to any interface, and has no effect on network operation.Files
